# Lodestar — 目標達成ノートアプリ 要件定義書（v0.3 ドラフト）

> **アプリ名: Lodestar（ロードスター／「導きの星」）**
> 由来: 目標へ導く星。STAR PLANNER の星モチーフと「目標達成の指針」を一語で表す。

> 原田メソッド「STAR PLANNER（目標達成ノート）」をスマホで使えるよう
> デジタル化する、個人専用の PWA（Web アプリ）。
> このドキュメントは Claude Code に開発を依頼するための出発点（仕様の土台）です。

---

## 0. このドキュメントの使い方

- 「✅確定」は質問への回答で確定済み。「🔧提案（要確認）」は私（開発者＝あなた）が後で決める想定の暫定案。

- **デプロイ方針: 案A（静的ホスティング）優先で進め、案B（クラウド同期）へ後から拡張できる設計を保つ**（詳細は第9章）。

## 1. 背景・目的

- 紙の手帳「STAR PLANNER」を毎日持ち歩かなくても、スマホで同じ習慣形成サイクルを回せるようにする。
- 紙の良さ（書く行為・一覧性）を損なわず、デジタルの強み（自動集計・リマインド・検索・連続記録の可視化）を足す。
- 最終ゴールは「毎日の記入が習慣化し、ルーティンの○が並ぶ様子と自信の積み上がりを実感できること」。

## 3. 用語整理（原田メソッド＝自立型セルフマネジメントの4ツール）

1. **オープンウィンドウ64** … 8×8＝64マスのマンダラチャート。未来思考編／実践思考編の2種。
2. **スターシート（目標・目的設定用紙）** … 1年後の目標を具体行動まで落とし込む用紙（STEP1〜5）。
3. **ジャーナル（日誌）** … 日々の記録。「今日のよかったこと」「今日をもう一度やり直せるなら」など。
4. **ルーティンチェック表** … 毎日やる行動の実行を○で記録。3週間（21日）連続で習慣化とみなす。

## 4. 機能要件

### 4.1 目標設定ツール

#### (A) オープンウィンドウ64
- **未来思考編**
  - 中央ブロック中心に「自分の未来」、周囲8マスに8ジャンルを配置。
    - 8ジャンル: 自分のこと（理想の自分）／仕事／健康（心・体）／趣味（レジャー・レクリエーション）／教養（教育・勉強）／収入・財産・老後／家族・家庭／奉仕活動（社会・他者のために）
  - 各ジャンルを外周ブロックの中心に置き、真上から時計回りに8つの目標を記入 → 計64マス。
  - 上部「スターゾーン」に〈夢の実現で得られる感情〉を記入。
  - マスは空欄のまま保存可（空欄＝成長ポイントとして許容）。
- **実践思考編**
  - 中央に「1年後までに達成したい目標」、周囲8マスに必要要素（基礎思考）、さらに各要素の行動目標（実践思考）。
  - 数字・期日を入れられる前提のマス。
- 共通要件
  - 64マスを**スマホで操作可能**にする（ピンチズーム or 9ブロックをタップで拡大して個別編集、が現実的）。
  - 1枚だけでなく**テーマ別に複数枚**作成・保存できる（仕事用／プライベート用など）。
  - 記入日を保持。

#### (B) スターシート（STEP1〜5）
- STEP1: 「家で毎日する奉仕活動」「職場で毎日する奉仕活動」
- STEP2: 4観点で目標を広げる … ①社会・他者−有形 ②社会・他者−無形 ③あなた自身−有形 ④あなた自身−無形（各複数行）
- STEP3: ポジティブセンテンス … 「私は ○年○月○日までに ○○をする！」形式（プラス感情の言葉を含める）
- STEP4: ①目標達成のため毎日行うルーティン ②目標達成までの行動・期日（マイルストーン）
- STEP5: 必要なストローク … 「誰から」「どんなストロークを」受け取りたいか
- 完成したスターシートは**いつでも参照できる場所**（ホーム等）に常設し、日々の記入と比較できる。

### 4.2 月次／週次／日次の記入フォーマット（紙の構成を踏襲）

紙の1ヶ月分の構成（あなたの確認に基づく）:

| パート | 紙のページ数 | アプリでの扱い |
|---|---|---|
| 月間カレンダー | 1ページ | 月ビュー（今月やること／メモ欄を併設） |
| スケジュール&ジャーナル欄 | 5ページ分（週単位） | 週ビュー（最大5〜6週）＋日次入力 |
| 今月の気付き／今月習慣化できたルーティン行動 | 2ページ見開き | 月次ふりかえりビュー |

- **月間カレンダー**: 月〜日のマス、「今月やること」、MEMO。日付タップで日次へ遷移。
- **週ビュー（スケジュール&ジャーナル）**: 「今週のタスク」「今週感じたいプラス感情」「今週のルーティン行動リスト」、各日の予定。
- **日次ジャーナル**: ①今日のタスク ②ルーティンチェック（○/×） ③今日のよかったこと ④今日をもう一度やり直せるなら（改善行動）。
- **月次ふりかえり**: 「今月の気付き」「今月習慣化できたルーティン行動（日数）」。

### 4.3 ルーティンチェックと習慣化ロジック
- ルーティン行動を登録 → 毎日 ○/× を記録。
- **連続○日数**を自動カウント。**21日（3週間）連続**で「習慣化」と判定し、お祝い表示＋「リストから外して新ルーティンを追加」を提案。
- 習慣化した項目はアーカイブして履歴に残す（達成実績として自信の可視化に使う）。
- 挫折対応: 連続が途切れても**累計実行日数**は保持・表示（「断続的でも継続は継続」の思想を反映）。

### 4.4 補助機能（あると効果が高いもの）🔧提案
- **プラス感情ワード集**: 記入時に候補から選べる語彙リスト（手帳 P.14 相当）。自分の言葉を追加可。
- **デイリーリマインド通知**（PWA の通知。端末対応次第）。
- **検索**（過去のよかったこと・気付きを横断検索 → 自信回復に使う）。
- **データのエクスポート/インポート（JSON）**: バックアップと機種変更対策。

## 6. データモデル（暫定）🔧提案

```
Settings            : テーマ, 通知設定, 開始日 など
EmotionWord         : id, label, isCustom
OpenWindow64        : id, type(future|practice), title, createdAt,
                      star(感情リスト), cells[9ブロック×9マス]
StarSheet           : id, title, createdAt,
                      step1(home, work), step2(4観点), step3(positiveSentence),
                      step4(routines[], milestones[]), step5(strokes[])
Routine             : id, name, emotionTag, createdAt,
                      status(active|archived), archivedAt
DailyEntry          : date(YYYY-MM-DD), tasks[], good(よかったこと),
                      redo(やり直せるなら), routineChecks[{routineId, done}]
WeeklyEntry         : weekKey, weekTasks[], targetEmotions[]
MonthlyReflection   : monthKey, insights(気付き), habituatedRoutines[]
```
- 連続日数・累計日数は DailyEntry から算出（保存値ではなく計算で求める）。

**案B（同期）に備えた共通設計** — 案Aでは未使用だが最初から入れておく:
- 全レコードに `id`（UUID）／`createdAt`／`updatedAt`／`deleted`（論理削除フラグ）を持たせる。
  → 将来の同期・差分マージ・競合解決（レコード単位の last-write-wins）に必要。
- 物理削除はせず `deleted=true` にする（同期時に削除を伝播できるようにするため）。
- 端末ローカルに `lastSyncedAt` を1つ保持（案Aでは null のまま）。

## 7. 非機能要件

- **PWA**: manifest＋Service Worker。ホーム画面追加、オフライン起動。
- **保存**: 端末内 DB（IndexedDB）。サーバー送信なし＝プライバシー上も安全。
- **バックアップ**: JSON エクスポート/インポート必須（ローカル保存はクリアで消えるため）。
- **レスポンス**: 記入はオフラインでも即保存。
- **アクセシビリティ**: 文字サイズ・コントラストに配慮。片手操作しやすい配置。
- **デプロイ**: AWS S3（非公開）+ CloudFront（OAC 経由）で静的配信。HTTPS 必須（PWA 要件）。
- **拡張性**: データアクセスは抽象レイヤー（Repository インターフェース）越しに行い、保存先を「ローカル（IndexedDB）」から「リモート（案BのAPI）」へ差し替え可能にする。UI 層は保存先を意識しない。
- **コスト**: 常時無料枠の範囲で実質 $0／月を維持（目標 月$5以下）。AWS Budgets で少額アラートを設定。

## 8. 技術スタック提案 🔧提案

| 層 | 採用案 | 理由 |
|---|---|---|
| UI | React + TypeScript | Claude Code が扱いやすく情報量も多い |
| ビルド | Vite | 軽量・PWA プラグインが充実 |
| スタイル | Tailwind CSS | スマホUIを素早く調整しやすい |
| ローカルDB | IndexedDB（Dexie.js） | 大量の日次データに耐える・クエリしやすい |
| PWA | vite-plugin-pwa | manifest/SW を自動生成 |
| ホスティング | **AWS S3 + CloudFront（案A）** | サーバー不要・常時無料枠で実質$0（第9章） |

- バックエンドなしの**フロントエンド完結型**を案Aの基本とする（自分専用＋オフライン要件に最適）。
- **拡張性の原則**: データ層を Repository インターフェースで抽象化し、案Aは `LocalRepository`（IndexedDB 実装）のみ。案Bでは同じインターフェースの `RemoteRepository`（API 実装）を足すだけで UI を変えずに同期対応できる。
- API のベースURLは環境変数（例: `VITE_API_BASE_URL`）で持ち、案Aでは空。案Bで値を入れると同期が有効になる作りにする。
- JSON エクスポート/インポートは案A時点のバックアップ手段であると同時に、案Bへの移行ブリッジにもなる。

**デザイン方針（✅確定: 水色×ミニマル踏襲）**:
- 白基調＋水色（STAR PLANNER 風の淡いブルー）をアクセントに。余白を広めに取り、装飾は最小限。
- 見出しは英語の大きな数字＋「○○ MONTH」など、元手帳の雰囲気を軽く参照（やりすぎない）。
- フォントは可読性優先のサンセリフ。タップ領域は大きめ（片手操作）。
- 具体的な配色・余白・タイポは実装時にデザイントークン（CSS 変数 / Tailwind 設定）として定義。

## 9. AWS デプロイ構成（案A優先・案B拡張）

### 9.1 案A（優先・まず作る）— 静的ホスティングのみ

```
[ブラウザ/PWA] ──HTTPS──> [CloudFront] ──OAC──> [S3（非公開バケット）]
        └─ データは端末内 IndexedDB（サーバー送信なし）
```

- **S3**: ビルド成果物（HTML/JS/CSS）を格納。バケットは非公開にし、CloudFront の OAC（Origin Access Control）経由でのみ配信。
- **CloudFront**: HTTPS 配信・キャッシュ・独自ドメイン対応。SPA 用に 403/404 を `index.html` へフォールバック。
- **証明書**: ACM（us-east-1）で無料の TLS 証明書。
- **DNS**: ✅確定で**独自ドメインは使わない**。CloudFront 既定ドメイン（`*.cloudfront.net`）を利用＝Route 53 不要・登録料ゼロ。（将来ドメインを足す場合のみ Route 53 を追加）
- **デプロイ手順**: `vite build` → S3 へ同期（`aws s3 sync`）→ CloudFront キャッシュ無効化（`create-invalidation`）。Claude Code には簡単なデプロイスクリプト（npm script）も作らせる。
- **IaC（推奨）**: AWS CDK か CloudFormation で S3/CloudFront/ACM を定義し、再現可能に。

**コスト（案A）**: CloudFront は無料枠（月1TB転送・1000万リクエスト、恒久）。S3 は数MB＝ほぼ無料。独自ドメインなしのため登録料も不要 → **完全 $0／月**（目標の月$5以下を余裕で達成）。

### 9.2 案B（将来拡張）— サーバーレス同期を足す

案Aの静的配信はそのまま残し、データ同期用のバックエンドを追加する。

```
[PWA] ──> [CloudFront] ──> [S3]（フロントは案Aのまま）
   └──> [API Gateway(HTTP API)] ──> [Lambda] ──> [DynamoDB]
                                       └─ 認証（単一ユーザー）
```

- **API Gateway(HTTP API) + Lambda + DynamoDB**: 単一ユーザーなら Lambda（月100万リクエスト）/ DynamoDB（25GB・約2億リクエスト相当）の常時無料枠に収まり、実質 $0〜1／月。
- **認証**: 個人専用なので軽量で可（事前共有トークン＋HTTPS、または Cognito ユーザー1名）。秘密情報はコードに直書きせず環境変数/設定で扱う。
- **同期方式**: レコード単位の差分同期（`updatedAt` 比較＋ last-write-wins、`deleted` 伝播）。第6章の共通フィールドがそのまま効く。
- **データモデル**: DynamoDB は1テーブル設計（PK=エンティティ種別、SK=id 等）で十分。

**コスト（案B）**: 無料枠内で $0〜1／月＋ドメイン代。$5 以内を維持。

### 9.3 案A→案B をスムーズにするための設計ルール（重要）

1. **データ層を抽象化**: UI は `Repository` インターフェース越しにのみ読み書き。案Aは `LocalRepository`(IndexedDB) のみ実装、案Bで `RemoteRepository`(API) を追加。
2. **同期メタ情報を最初から保持**: 全レコードに `id`/`createdAt`/`updatedAt`/`deleted`（第6章）。論理削除を徹底。
3. **API URL は環境変数**: `VITE_API_BASE_URL` が空なら案A（ローカルのみ）、値があれば案B（同期有効）。
4. **JSON 入出力を移行ブリッジに**: 案Aで貯めたデータは JSON エクスポート→案Bの初回アップロードで引き継げる。
5. **フロントは案Bでも不変**: S3+CloudFront 配信は維持。追加するのは API/DB レイヤーのみ。

### 9.4 コスト面の注意（$5 を超えないために）

- 新規 AWS アカウントは **Paid Plan（従量課金）を選ぶ**。6か月で自動閉鎖される Free Plan は長期運用に不向き。常時無料枠の範囲で実質$0を維持する。
- **EC2 / NAT Gateway / 公開IPv4 は使わない**（NAT は月約$32、公開IPv4は月約$3.60と高額で無料枠外）。静的＋サーバーレスで通す。
- AWS Budgets で月 $1〜5 のアラートを設定。
- AWS の料金・無料枠は変動が大きいため、着手前に公式料金ページで最新値を確認する。

## 10. 開発フェーズ（Claude Code での進め方）🔧提案

- **Phase 0 — 土台**: Vite+React+TS+Tailwind 雛形、ボトムナビ、PWA 設定、**Repository 抽象レイヤー＋ LocalRepository(IndexedDB) 実装**（案B拡張の起点）。
- **Phase 1 — 日次の核**: 日次ジャーナル＋ルーティンチェック（毎日使う最重要部分を最優先）。
- **Phase 2 — ルーティン習慣化**: ルーティン登録・連続/累計カウント・21日習慣化判定・アーカイブ。
- **Phase 3 — 週/月ビュー**: スケジュール&ジャーナル週ビュー、月間カレンダー、月次ふりかえり。
- **Phase 4 — 目標設定ツール**: スターシート（STEP1〜5）→ オープンウィンドウ64。
- **Phase 5 — 仕上げ**: 感情ワード集、検索、通知、JSONバックアップ、デザイン調整。
- **Phase 6 — デプロイ（案A）**: S3+CloudFront 構成を IaC 化、ビルド→同期→キャッシュ無効化のデプロイスクリプト整備。
- **（将来）Phase 7 — 案B同期**: API Gateway+Lambda+DynamoDB と RemoteRepository を追加。フロントは原則そのまま。

> ポイント: 紙を全部再現してから使い始めるのではなく、**毎日触る Phase 1〜2 を先に動かして習慣化を始める**のが続けるコツ（手帳の思想とも一致）。
